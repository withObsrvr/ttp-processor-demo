Hi everyone,
We are currently building a PoC and we’ve hit an infrastructure roadblock regarding cross-chain bridging into Soroban.
Our Architecture / Flow:
1. User initiates from a Crossmint Smart Account on Base.
2. Bridge USDC to a Crossmint Smart Account on Stellar (Soroban).
3. The Stellar Smart Account then interacts with Defindex.
The Problem:
We’ve been testing the bridging layer using Allbridge and Near Intents, but both confirmed they do not support bridging directly to a Soroban Smart Contract address. Their systems still rely on classic Stellar account architecture and look for traditional manual trustlines before releasing funds, which obviously fails or isn't supported when the destination is a Smart Contract
My questions are:
1. Are there any bridges currently live that support sending assets directly to a Soroban Smart Contract address?
2. If direct bridging isn't supported yet, what is the recommended design pattern for this flow? Are devs currently bridging to a classic Stellar account first, and then using a relayer/bot/contract to forward the funds to the Smart Account and execute the next steps?

Technically, the first step is **not** “become the ramp.” It is to support the pattern that lets regulated ramps and bridges **land assets into a Soroban-aware destination safely**.

What the protocol already gives you is important:

Stellar’s **Address** type can represent a classic account, a custom Soroban account, or an arbitrary contract address, so Soroban is already designed around more than classic `G...` accounts. ([developers.stellar.org][1])

Also, the **Stellar Asset Contract (SAC)** explicitly supports assets being held by both **accounts and contracts**. Stellar’s docs say the SAC can interact with assets held by Stellar accounts **or contracts**, and that transfers between an account and a contract debit the account trustline and credit contract data, or vice versa. ([developers.stellar.org][2]) Once the SAC for an asset is deployed, any contract can hold, receive, and send that asset. ([developers.stellar.org][3])

So the blocker here is not Soroban itself. The blocker is that bridge/ramp implementations and wallet abstractions still often assume a **classic-wallet deposit model**. That lines up with what you saw from NEAR Intents: their Stellar support is documented in terms of **SEP-53 Stellar wallets**, not Soroban smart-account destinations. ([docs.near-intents.org][4]) Crossmint’s Stellar smart account is also clearly a Soroban smart-contract account system, not a classic account wrapper. ([GitHub][5])

My read is this:

**Direct bridge-to-Soroban-smart-account support is still immature or inconsistent.** I did not find strong evidence from current public docs that Allbridge or NEAR Intents broadly support “send directly to arbitrary Soroban smart-account/contract address and complete delivery there” as a standard production flow. What I *did* find is that Allbridge has Soroban-native bridge contracts and plumbing on Stellar, which shows the chain-side pieces exist, but that is different from universal destination support for smart-account addresses. ([allbridge.io][6])

Because of that, the best current design pattern is usually:

1. **Bridge into a classic Stellar account you control**
2. Ensure the needed asset’s **SAC is deployed**
3. Have a **relayer/orchestrator** submit a Soroban invocation that moves funds from that landing account into the smart account / contract
4. In the same transaction or next step, call Defindex

That is the most practical first implementation because it works with today’s bridge assumptions while still ending in the Soroban-native flow.

My opinion is that the **first thing to support** for this ecosystem problem is a **Soroban-aware settlement adapter**, not a full regulated on/off ramp. Concretely, that means building:

* a **landing-account pattern** for each supported asset/network
* an **asset/SAC registry** so you know whether the asset can be forwarded contract-to-contract
* a **forwarder/relayer service** that watches the landing account, validates arrival, and invokes the smart-account/Defindex flow
* optional **memo / intent correlation** so a bridge deposit can be matched to a specific Soroban execution request

That becomes valuable infrastructure for regulated ramps, wallets, and bridges without you being the regulated fiat endpoint yourself.

If I were defining the “first step” product-wise, I would make it:

**“Soroban Deposit Forwarder”**

* Input: bridged funds arrive at a classic Stellar settlement account
* Logic: detect deposit, verify asset + amount + source mapping
* Action: invoke SAC transfer into the Soroban smart account, then call the target protocol
* Output: one reliable bridge-to-Soroban execution path

That solves the immediate pain in the Discord post.

**1. Are there live bridges that support sending directly to a Soroban smart contract address?**

**2. What is the recommended design pattern right now?**
Yes: **bridge to a classic account first, then forward via a relayer/bot/contract into the smart account and continue execution.** That is the safest near-term pattern because it matches current bridge expectations while still using Soroban correctly.

For Obsrvr specifically, this could turn into a real support surface:

* resolve asset-to-SAC mappings

That is a very plausible first infrastructure product around ramps without touching the regulated fiat edge yourself.

# What the RFP Is Actually Asking For

The problem statement is deceptively simple:

> “Fund a C-address without using a G-address.”

But technically, this is not really possible at the protocol layer today.

Soroban **C-addresses are contracts**, not accounts in the classic ledger model. They cannot:

* receive **classic Stellar payments**
* hold **trustlines**
* receive **CEX withdrawals**
* receive **bridge transfers**

because almost all infrastructure assumes a **G-address** destination.

So the ecosystem needs an **abstraction layer** that hides this limitation.

The real requirement is:

**Create a system where users think they are funding a C-address, even though the funds land in a temporary G-account and are forwarded automatically.**

In other words:

> **Make the G-address invisible.**

This is exactly the architecture pattern we discussed earlier with relayers.

---

# The Architecture Stellar Is Expecting

The most realistic architecture looks like this.

```
User / CEX / Bridge
        │
        │ withdraw to deposit address
        ▼
Temporary G Address (deposit account)
        │
        │ detected by service
        ▼
Forwarder Service / Relayer
        │
        │ Soroban invocation
        ▼
C Address (Smart Account)
        │
        ▼
Target protocol (DeFindex etc)
```

From the user's perspective:

```
Send → C Address
```

But behind the scenes:

```
Send → Deposit Account → Smart Account
```

This is **exactly how many rollups and smart wallets already work**.

## What I would change

I would organize Soroban around **three separate concerns** instead of one flat “Soroban support” bucket:

### 1. Execution

This is “what happened in a transaction.”

Keep raw transaction and operation facts in Bronze, because Hubble’s Bronze `history_transactions` already exposes Soroban fee/resource fields like `resource_fee`, `rent_fee_charged`, and instruction/read/write byte counts. `history_operations` also carries Soroban function details and decoded parameters. ([developers.stellar.org][5])

But in Silver, create a first-class execution layer:

* `contracts.invocations`
* `contracts.invocation_edges`
* `contracts.authz`
* `contracts.tx_execution_summary`

This is where Obsrvr can be much better than Hubble. Your own docs already position call graphs, caller/callee analysis, and function-level analytics as the differentiator. 

### 2. State

This is “what storage exists now, and how did it change.”

Hubble Bronze `contract_data` is useful, but it is still fundamentally a raw storage-entry table with keys, values, durability, ledger change flags, and XDR/decoded fields. Its natural key is `ledger_key_hash, closed_at`, partitioned by `closed_at`, and clustered by `last_modified_ledger, contract_id`. ([developers.stellar.org][6])

That is good for raw fidelity, but not ideal as your main Soroban analytic surface.

For Silver, I would separate:

* `contracts.storage_current`
* `contracts.storage_snapshot`
* `contracts.storage_changes`
* `contracts.ttl_current`
* `contracts.ttl_changes`
* `contracts.code_current`

This matters because Soroban state is not just “data rows.” TTL, restoration, eviction, and code lifecycle are part of the actual product surface. Hubble Bronze has dedicated tables for TTL, restored keys, and evicted keys, which is a clue that Soroban state needs lifecycle modeling, not just a generic storage table. ([developers.stellar.org][7])

### 3. Asset movement

This is “who got paid, in what token, by what mechanism.”

This is where many schemas get messy. If you rely only on classic `history_operations`, Soroban token movement becomes harder to reason about.

Hubble already moves in the right direction with `token_transfers_raw` in Silver, which normalizes transfers and includes `contract_id`. ([developers.stellar.org][4])

So for Obsrvr, I would make token movement a separate Soroban-ready domain:

* `assets.token_transfers`
* `assets.token_balances_current`
* `assets.token_holders_current`
* `assets.token_holders_daily`

That gives you one unified model for:

* classic payments
* path payment outcomes
* SAC transfers
* smart account funding flows
* DEX / protocol movements

## The structural mistake I would avoid

I would **not** let `contract_data` become your main Soroban table.

That table is necessary, but by itself it leads people into low-level key/value archaeology. It is a storage substrate, not a product model.

If Obsrvr Lake wants to be Soroban-first, the user should more naturally query:

* contract invocations
* function calls
* event emissions
* state changes
* token transfers
* smart account activity
* contract relationships

not just:

* raw operations
* raw events
* raw contract_data rows

Your own standards doc is already pointing there. It recommends adopting Hubble’s core schema structure, but then differentiating with real-time streaming, Soroban event decoding, and Gold-layer Soroban analytics like invocation volume, top contracts by gas, and smart wallet activity.

Because they are **raw substrate tables**, not user-facing analytics models. They preserve:

This also aligns with your Hubble compatibility work, which treats `contract_events` and `contract_data` as the core Soroban foundation.

# The most important structural decision

If I had to give one opinionated recommendation, it would be this:

## Do not make `contract_data` the center of the Soroban model

It should exist, but it should not be the main interface people think through.

Instead, organize Soroban around four first-class questions:

### 1. What executed?

* `contract_invocations_*`
* `contract_call_edges_*`
* `contract_failures_*`

### 2. What emitted?

* `contract_events_*`
* `contract_events_decoded_*`

### 3. What state changed?

* `contract_storage_changes_*`
* `contract_data_current_*`
* `contract_ttl_*`

### 4. What value moved?

* `token_transfers_*`
* `token_balances_*`
* `token_holders_*`

That is how you get from “raw contract support” to “Soroban analytics platform.”

# Obsrvr Lake — Soroban Data Model Specification

## Overview

Obsrvr Lake implements a **medallion architecture** for Stellar and Soroban data:

```
Bronze → Silver → Gold
```

| Layer  | Purpose                                      |
| ------ | -------------------------------------------- |
| Bronze | Raw ledger ingestion and immutable snapshots |
| Silver | Canonical normalized domain tables           |
| Gold   | Aggregated analytics datasets                |

This structure provides:

* **Hubble compatibility**
* **Soroban-native analytics**
* **API-ready datasets**
* **long-term schema stability**

---

# Naming Convention

All datasets follow the convention:

```
<domain>.<dataset>_<grain>_v<version>
```

Example:

```
contracts.contract_invocations_enriched_v1
assets.token_transfers_daily_v1
```

### Dataset components

| Component | Meaning                                               |
| --------- | ----------------------------------------------------- |
| domain    | logical grouping (`core`, `contracts`, `assets`, etc) |
| dataset   | subject of the table                                  |
| grain     | level of aggregation (`row`, `daily`, `current`)      |
| version   | schema version                                        |

## What this layer is

It is not “another copy of the data.”

It is a **contract of meaning**:

* canonical entities
* reusable business logic
* stable metrics definitions
* intent-based query surfaces
* graph and state abstractions

In other words, instead of forcing every consumer to know:

* which Silver tables to join
* how Soroban state should be reconstructed
* how to interpret smart accounts
* how to unify classic payments and SAC transfers

you give them a small set of **first-class concepts**.

That fits your own architecture philosophy very well, because your docs already position Silver as the canonical source of truth and Gold as the delivery layer for insights.  And your stated differentiation is not just raw access, but **intentional data products** that people can understand and build on. 

## Why this simplifies the schema

Right now, a Soroban-capable schema tends to explode into many tables:

* invocations
* events
* decoded events
* storage current
* storage history
* TTL current
* token transfers
* smart accounts
* fee payers
* call edges
* call trees

That is correct internally, but most consumers do not actually want tables. They want answers to questions like:

* What happened in this transaction?
* What contracts did this wallet use?
* What token moved?
* What state changed?
* What protocol action occurred?

Your current docs already point toward this style of product surface. Obsrvr Lake is meant to provide REST endpoints and analytics-ready datasets so users do not need complex SQL. # The Key Insight

You do **not** need to build all four layers immediately.

This is the real dependency chain:

```text
Bronze → Silver → Semantic → Graph → Gold
```

But in practice:

* Graph depends on Semantic
* Gold depends on Semantic
* APIs depend on Semantic
* Explorer depends on Semantic

So the **real keystone layer** is:

> **Semantic**

If you get that right, everything else becomes much easier.

# What You Should Build First

## Phase 1 — Silver (Minimal)

You already have much of this.

Goal: **canonical chain model**

Tables to prioritize:

```
silver.core.history_transactions
silver.core.history_operations
silver.contracts.contract_invocations_enriched
silver.contracts.contract_events_decoded
silver.assets.token_transfers_enriched
```

These five tables unlock almost everything.

Why?

Because they answer:

* what transactions happened
* what contracts were called
* what events emitted
* what tokens moved

That is the entire Soroban runtime.

You likely already have most of this from Flow ingestion.

---

# Phase 2 — Semantic Layer (Most Important)

This is where the real simplification happens.

Only build **three tables first**.

```
semantic.activities
semantic.entities_contracts
semantic.flows_value
```

These give you:

### activities

“What happened”

### entities_contracts

“What contracts exist”

### flows_value

“What value moved”

With those three tables, you can build:

* block explorer
* protocol analytics
* wallet analytics
* token analytics
* most APIs
